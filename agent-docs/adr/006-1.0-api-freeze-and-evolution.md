# ADR-006: 1.0 API freeze & evolution policy

## Status

Accepted (2026-06-29). Gates the 1.0 tag.

## Context

Nothing in the repo stated **what** the stable public API is, **what** SemVer
guarantees apply, or how deprecations work. By [Hyrum's Law](https://www.hyrumslaw.com/)
every incidental export (example/demo types, internal event helpers, namespace
internals) silently becomes a de-facto frozen contract the moment 1.0 ships. The
README's "Version support policy" covered only MUI driver majors, not the core or
the framework adapters. Three concrete contract gaps fed this decision and are
resolved alongside it:

- The published `@atomic-testing/playwright` depended at runtime on
  `@atomic-testing/internal-test-runner` (branded _"experimental … not for
  production use"_) and re-exported its types — issue #957 (B1).
- A missing element threw a **different error class per environment**
  (`ElementNotFoundError` in jsdom, Playwright's `TimeoutError` in the browser) —
  issue #958 (B2).
- React adapters advertised peer ranges their dependency graphs could not satisfy
  — issue #959 (B3).

This ADR is the deliberate place those decisions are ratified, because the 1.0
freeze forces them to be made on purpose rather than inherited by accident.

## Decision

### 1. The frozen public surface

The stable 1.0 API is **exactly the `.` barrel exports of the in-scope packages**,
and nothing else:

`core`, `dom-core`, `react-core`, `react-18`, `react-19`, `react-legacy`, `vue-3`,
`playwright`, `component-driver-html`.

Everything else — every `@atomic-testing/internal-*` package, every non-`.`
import path, and every export tagged `@internal` — is **not** covered by the 1.0
guarantee and may change in any release.

The published MUI and Astryx driver packages (`component-driver-mui-*`,
`component-driver-mui-x-*`, `component-driver-astryx`) are deliberately **out of
this freeze**: they track fast-moving third-party DOM and are governed by the
per-major support policy ([ADR-003](003-version-specific-packages.md),
[ADR-005](005-drop-mui-5-support.md)), not the core 1.0 contract.

The surface is **machine-checked**: each in-scope package commits an API report
at `etc/<package>.api.md` generated by [API Extractor](https://api-extractor.com/),
and the `check:api` gate (CI job _API surface freeze_) fails the build if a
package's exported surface drifts from its committed report. Adding, removing, or
changing the signature of any export is therefore a visible, reviewed diff — the
report **is** the contract. Regenerate intentionally with `api-extractor run
--local` per package (or after a `pnpm -r build`).

Incidental exports are **demoted, not removed** (removal would itself be the
breaking change the freeze exists to prevent): they keep their current names but
carry an `@internal` TSDoc tag, which the report records. The first demotions are
`IExampleUIUnit`/`IExampleUnit` (`core`) and `FakeMouseEvent` (`dom-core`).

### 2. SemVer policy & deprecation lifecycle

Once 1.0 ships, the in-scope packages follow [SemVer](https://semver.org/) against
the frozen surface:

- **patch** — bug fixes, no surface change.
- **minor** — additive only (new exports, new optional parameters, new optional
  `Interactor` capabilities); never removes or narrows existing surface.
- **major** — the only release allowed to remove or break frozen surface.

Deprecation lifecycle: a member is first marked `@deprecated` (TSDoc, so it
surfaces as an editor/compiler hint — the warning channel), **survives at least
one minor release** in that state, and is only removed at the **next major**. This
mirrors the deprecate-in-place precedent set for MUI 5 in
[ADR-005](005-drop-mui-5-support.md): keep the old thing installable and warning
while consumers migrate.

### 3. Support matrix (part of the contract)

The advertised peer ranges are part of the 1.0 contract and must each install a
coherent tree:

| Adapter        | React / Vue                      | Notes                                                                                                                                          |
| -------------- | -------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| `react-18`     | React `>=18 <19`                 |                                                                                                                                                |
| `react-19`     | React `>=19`                     |                                                                                                                                                |
| `react-core`   | React `>=18`                     | Shared base; uses `act` from `@testing-library/react@16` (peer React 18/19). Corrected from a dishonest `>=16` floor (B3).                     |
| `react-legacy` | React `^16 \|\| ^17`             | Self-contained `act` via `react-dom/test-utils`; deliberately off the `@testing-library/react@16` graph so the React 16/17 tree resolves (B3). |
| `vue-3`        | Vue `3.x`                        |                                                                                                                                                |
| `playwright`   | `@playwright/test >=1.50` (peer) | Browser driver only; see boundary below.                                                                                                       |

Node `>=22.12`, pnpm `>=10` for development (per root `package.json` `engines`).

### 4. Internal vs. stable boundary (resolves B1)

No shipped 1.0 package depends at runtime on, or re-exports types from, a package
branded `internal`/`experimental`. The shared test-runner tooling
(`internal-test-runner` and its jest/vitest/playwright adapters), the example
harness, and the MUI-X fixture are all `private: true` and never published; the
Playwright test-runner glue lives in `internal-test-runner-playwright-adapter`
(which depends on `playwright`, inverting the old edge). `publish.sh` honors the
`private` flag — private packages still build (the repo's own tests consume their
`dist`) but are never published.

### 5. Error contract (resolves B2)

Every **mutative** `Interactor` method throws `ElementNotFoundError` when its
locator matches nothing, in **every** environment. `PlaywrightInteractor`
translates Playwright's auto-wait `TimeoutError` into `ElementNotFoundError` only
when the element is genuinely absent, otherwise rethrowing — so a shared
`*.suite.ts` can assert the error without branching on the environment. This is
the contract for the frozen `Interactor` interface; each method documents it in
TSDoc.

### 6. Evolving the `Interactor` post-1.0

Adding a method to the `Interactor` interface is a **breaking change for external
implementers** ([ADR-002](002-interactor-abstraction.md) already notes that
`PlaywrightInteractor` does not inherit new methods from `DOMInteractor`). Under
this policy, post-1.0 `Interactor` growth must be **additive without breaking
implementers** — e.g. optional methods, a default-providing base, or capability
detection rather than a bare required-method addition. The concrete mechanism is
deferred to **D1** (the Interactor-evolution issue); this ADR fixes the
constraint, not the implementation.

## Consequences

- ✅ The public surface is explicit, reviewed, and CI-enforced; incidental
  exports can no longer be frozen by accident.
- ✅ Consumers get a real SemVer + deprecation guarantee, and a support matrix
  that actually installs.
- ✅ No 1.0 package carries an `internal`/`experimental` runtime dependency.
- ⚠️ Every intentional surface change now requires regenerating and committing the
  affected `etc/*.api.md` report — a small extra step, and the point.
- ⚠️ The `etc/*.api.md` reports retain `ae-forgotten-export` notes where a public
  type references a non-exported symbol (e.g. `CssLocator` → `CssLocatorSource`).
  These are surfaced deliberately as cleanup candidates; resolving them (export or
  inline) is follow-up work, not part of the freeze.
- ⚠️ API Extractor's bundled TypeScript trails the repo's `tsgo`/`typescript@6`;
  it prints a version-mismatch note but analyzes the bundled `.d.mts` correctly.

## Alternatives considered

| Alternative                                                                | Why not chosen                                                                                                                                                                          |
| -------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Document the policy in prose only, no tooling                              | Prose drifts; Hyrum's-Law freezes happen silently. The api-extractor report makes every surface change a reviewed diff.                                                                 |
| A bespoke zero-dependency surface snapshot (like `check-doc-accuracy.mjs`) | Reinvents API Extractor's signature-aware diffing; api-extractor also gives release-tag trimming and `.d.ts` rollups for free.                                                          |
| Freeze only `core`                                                         | The adapters (`playwright`, `react-*`, `vue-3`) and `component-driver-html` are the consumer entry points; freezing core alone leaves the surfaces consumers actually import unguarded. |
| Delete incidental exports outright                                         | Removal is itself the breaking change the freeze prevents; demote with `@internal` instead.                                                                                             |
