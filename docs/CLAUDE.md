# docs/CLAUDE.md

How the Atomic Testing documentation site is built, and how to verify it locally.
This covers the `docs/` Docusaurus site only; the library packages live under
`packages/` and follow the root [`CLAUDE.md`](../CLAUDE.md).

## Stack

Static site built with **Docusaurus 3.10** (`@docusaurus/preset-classic`), with the
mermaid theme, Algolia search, and `future.faster` builds enabled. The narrative docs
are hand-authored MDX under `docs/docs/`; the **API reference is generated from the
TypeScript source** of every documented package by `docusaurus-plugin-typedoc` (which
wraps TypeDoc + `typedoc-plugin-markdown`) at build/start time. Config lives in
[`docusaurus.config.ts`](docusaurus.config.ts); dependencies and scripts in
[`package.json`](package.json).

**Toolchain divergence from the repo:** the root build type-_checks_ with `tsgo`
(TypeScript 7 preview), but TypeDoc here runs on the **classic `typescript@^6`
compiler** (pinned in `package.json` alongside `typedoc`), because TypeDoc has no
tsgo backend. So the API reference is only as current as what classic `tsc` reads
from the package sources.

## How the API reference is generated & assembled

The reference is a pipeline of small, single-purpose pieces; each is the source of
truth for one decision, so change the concept at its owner rather than patching
downstream.

- [`scripts/docPackages.mjs`](scripts/docPackages.mjs) — the **single source of truth**
  for which `packages/*` are documented and how they're bucketed (framework/interactor
  vs. `component-driver-*`). It reads `packages/` on disk directly (not the generated
  tree, which doesn't exist yet when config is evaluated), dropping dot-dirs, `internal-*`
  helpers, and anything in `EXCLUDED_FROM_DOCS`. Both the config and the sidebar import
  from here, so the package list can't drift between them.

- [`docusaurus.config.ts`](docusaurus.config.ts) — the `docusaurus-plugin-typedoc`
  block drives TypeDoc: `entryPointStrategy: 'packages'` (one entry point per documented
  package), `excludeInternal` (hides `@internal`-tagged exports), `excludeScopesInPaths`
  (emits `api/core` rather than nesting under an `api/@atomic-testing/` scope dir),
  `readme: 'none'`, and the custom `theme: 'markdown-partitioned'` (see below). The
  classic preset's `sidebarItemsGenerator` then **reshapes the autogenerated API sidebar**
  via `shapeApiPackage`: TypeDoc emits each package as lowercase per-kind folders with
  verbose, MDX-escaped labels (`Abstract Class: ComponentDriver\<T\>`); `shapeApiPackage`
  wraps each `api/<name>` package as one collapsed category linked to its landing page,
  cleans the labels (drops the kind prefix, un-escapes generics), and shapes the interior
  by bucket — driver packages surface their driver classes flat with supporting kinds as
  collapsed subcategories, framework packages group every kind (Classes included) into its
  own subcategory. `shapeApiPackage` and [`sidebars.ts`](sidebars.ts) are the source of
  truth for the shape.

- [`sidebars.ts`](sidebars.ts) — the hand-authored `tutorialSidebar`. The
  **📖 API Reference** category is split into **Framework & Interactor APIs** vs.
  **Component Driver APIs** (using the buckets from `docPackages.mjs`), because the two
  read differently: framework packages are thin wiring around one `Interactor`, driver
  packages are a per-component vocabulary of many driver classes. The category uses the
  `/api-reference` slug deliberately — TypeDoc always writes its own flat `docs/api/`
  index, so reusing `/api` would collide.

- [`typedoc/partition-members-plugin.mjs`](typedoc/partition-members-plugin.mjs) — the
  custom TypeDoc markdown theme registered as `markdown-partitioned`. It splits each
  class/interface page's members into **own / inherited / protected** sections (own
  expanded, the rest collapsed), so a leaf driver surfaces the handful of methods a
  reader calls instead of interleaving them alphabetically with inherited base-driver
  and protected members. Classification reads TypeDoc reflection flags only, never
  comment tags.

## The generated `api/` tree is ephemeral

`docs/docs/api/` is produced by TypeDoc at build/start and is **gitignored** (the
repo-root [`.gitignore`](../.gitignore) entry `docs/docs/api`). There is no source file
for it in the repo — which is why `editUrl` in `docusaurus.config.ts` returns
`undefined` for `api/` paths, suppressing the "Edit this page" link that would otherwise
404. **Never hand-edit anything under `docs/docs/api/`**; it is regenerated on every
build and your edit is lost. Change the generating symbol instead (the package source,
the plugin options, or the partition theme).

## Doc-accuracy CI gates

Two dependency-free Node scripts run in CI to keep hand-authored docs honest against the
live package source:

- [`scripts/check-doc-accuracy.mjs`](scripts/check-doc-accuracy.mjs) — asserts every
  `@atomic-testing/*` symbol imported in the docs (including code embedded as template
  strings in `src/*.tsx`) is a real export of that package's `src/index.ts`, and that a
  denylist of known-fabricated names never appears. Catches inline snippets that were
  never compiled.

- [`scripts/check-doc-driver-sync.mjs`](scripts/check-doc-driver-sync.mjs) — asserts any
  package excluded from the generated reference (`EXCLUDED_FROM_DOCS` in
  `docPackages.mjs`) is never presented as a normal option in the narrative guides:
  every section of `framework-guide.mdx` that mentions such a package must carry an EOL
  marker. No-ops cleanly when nothing is excluded.

## Building & verifying locally

**The API reference needs the workspace packages built first.** TypeDoc reads each
package's `src`, but resolves the cross-package (`@atomic-testing/*`) and framework-peer
types those sources import from the built `dist`. On a fresh checkout `dist` doesn't
exist, so TypeDoc resolves nothing, aborts every package's conversion, and the entire
API reference comes out **empty** (which then cascades into broken-link failures for the
`/docs/api-reference/` nav link). CI does the right order in
[`doc-ci.yml`](../.github/workflows/doc-ci.yml): `./publish.sh --build-only` (build every
package) → `pnpm install .` in `docs/` → `pnpm build`. Reproduce that order locally.

From a fresh checkout, the simplest path is the root `package.json` scripts: `pnpm run
setup` installs both the monorepo and this standalone project, then `pnpm run docs:dev`
builds the packages and starts the dev server. The manual two-step order below remains
useful for offline/standalone installs of `docs/` alone.

`docs/` is **not part of the pnpm workspace** (`pnpm-workspace.yaml` covers only
`packages/*` and `package-tests/*`); it has its own [`pnpm-lock.yaml`](pnpm-lock.yaml)
and installs standalone. Per [`README.md`](README.md), install from `docs/` with
`pnpm install .`. On a machine with no npm-registry network but a warm global pnpm store,
install offline and standalone — `--ignore-workspace` is essential, or pnpm walks up to
the monorepo `pnpm-workspace.yaml` and tries (and offline, fails) to install all of it:

```bash
pnpm install --offline --ignore-workspace --store-dir ~/Library/pnpm/store
```

Then, all from `docs/`:

- `pnpm start` — dev server on **http://localhost:3333** (port set in the `start`
  script). The TypeDoc API tree is generated during startup, so the **first run is slow**
  (~28 packages); verify the API sidebar shape and any driver page here.
- `pnpm build` — production build into `docs/build/`, then `pnpm serve` to preview it.
  Run this before any docs PR — a broken link `throw`s the build.
- `pnpm typecheck` — `tsc` over the site's own TS (`docusaurus.config.ts`, `sidebars.ts`,
  `src/`).

## Documentation Freshness Rules

The root CLAUDE.md points contributors to each package's own freshness rules; these are
this site's.

- When you change **how the sidebar or API reference is assembled** —
  [`sidebars.ts`](sidebars.ts), the `sidebarItemsGenerator` in
  [`docusaurus.config.ts`](docusaurus.config.ts),
  [`scripts/docPackages.mjs`](scripts/docPackages.mjs), or the partition theme in
  [`typedoc/partition-members-plugin.mjs`](typedoc/partition-members-plugin.mjs) — update
  the matching description in this file.
- Keep those descriptions at **altitude**: name the symbol and its purpose, don't
  transcribe its options, buckets, or transform steps. Routine content changes (adding a
  package, retitling a category) should not require a doc edit here — only a change in how
  the pipeline works should.
