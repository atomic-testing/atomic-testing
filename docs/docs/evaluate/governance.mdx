# Governance & risk

This page exists so your due diligence finds nothing we hid. It leads with the
project's biggest adoption risk, then shows the machinery — not promises — that
mitigates it. Every claim links to a file you can open and check yourself.

## The headline risk: one maintainer, no SLA

Atomic Testing is maintained by a single person. The project's own
[SECURITY.md](https://github.com/atomic-testing/atomic-testing/blob/main/SECURITY.md)
says it plainly:

> Atomic Testing is currently maintained by a single maintainer on a best-effort
> basis. There's no guaranteed SLA, but reports are triaged and acknowledged
> within a few business days.

The [CODEOWNERS](https://github.com/atomic-testing/atomic-testing/blob/main/CODEOWNERS)
file is one line. You would find this on GitHub in thirty seconds; we would
rather you read it here first, alongside what it actually means for you.

The concrete risks of a single-maintainer dependency are: response latency on
bugs that block you, a bus-factor of one, and project direction that isn't set
by a committee you can join. What follows is the machinery that bounds each of
those risks — the things that keep working even on days no human is at the
keyboard.

## What mitigates it: machinery, not promises

A single maintainer can't promise responsiveness. What this project does
instead is encode its guarantees as CI gates and committed artifacts, so that
correctness doesn't depend on maintainer attention.

### The API surface is machine-frozen

Every stable-surface package commits an [API Extractor](https://api-extractor.com/)
report (e.g. [`packages/core/etc/core.api.md`](https://github.com/atomic-testing/atomic-testing/blob/main/packages/core/etc/core.api.md)),
and the `check:api` CI job fails any pull request whose exported surface drifts
from the committed report. An accidental breaking change cannot merge silently —
it shows up as a reviewable diff to the contract file. The full policy, the
exact frozen surface, and the SemVer/deprecation rules are on
[API stability, versioning & EOL](./api-stability.mdx).

### Portability is enforced by a conformance kit

The core claim — the same test suite runs unchanged under jsdom and real
browsers — is pinned by executable conformance suites, not by documentation.
If an interactor change breaks the contract, CI goes red. See
[Proof of portability](./proof-of-portability.mdx).

### Docs, recipes, and skills are drift-gated against source

Documentation that lies is worse than no documentation, so this site is checked
against the source it describes on every CI run:

- [`check-doc-accuracy.mjs`](https://github.com/atomic-testing/atomic-testing/blob/main/docs/scripts/check-doc-accuracy.mjs)
  — every `@atomic-testing/*` symbol imported in a docs code snippet must be a
  real export of that package's `src/index.ts`. Fabricated API names fail the
  docs build.
- [`check-doc-driver-sync.mjs`](https://github.com/atomic-testing/atomic-testing/blob/main/docs/scripts/check-doc-driver-sync.mjs)
  — an end-of-life driver package cannot be presented in the narrative docs as
  a normal, supported option without an EOL marker.
- [`check-support-matrix-sync.mjs`](https://github.com/atomic-testing/atomic-testing/blob/main/docs/scripts/check-support-matrix-sync.mjs)
  — the published [support matrix](../support-matrix.mdx) is regenerated from
  the scaffolder's compatibility registry and compared byte-for-byte with the
  committed copy, so a hand-edited or stale matrix can't ship.
- The scaffolder's own registry is gated the same way:
  [`check-scaffolder-coverage.mjs`](https://github.com/atomic-testing/atomic-testing/blob/main/packages/create-atomic-testing/scripts/check-scaffolder-coverage.mjs)
  fails CI if a shipped driver package isn't registered with the CLI, and
  [`check-recipe-sync.mjs`](https://github.com/atomic-testing/atomic-testing/blob/main/packages/create-atomic-testing/scripts/check-recipe-sync.mjs)
  fails CI if the registry would scaffold a package that doesn't exist, at an
  out-of-line version, or with a framework range the real engine peer doesn't
  satisfy.
- The agent skills that ship with the project are governed by
  [`scripts/skills/`](https://github.com/atomic-testing/atomic-testing/tree/main/scripts/skills)
  (`check:skill-sync`, `check:golden-fixtures` in CI), so skill claims can't
  silently drift from the code they describe.

### The changelog writes itself

`CHANGELOG.md` is generated by
[`scripts/generateChangelog.js`](https://github.com/atomic-testing/atomic-testing/blob/main/scripts/generateChangelog.js)
from the commit-subject convention documented in
[CONTRIBUTING.md](https://github.com/atomic-testing/atomic-testing/blob/main/CONTRIBUTING.md).
Release notes are a mechanical function of merged history, not a manual step
that a busy maintainer can skip or embellish.

### Releases can't leak a credential that doesn't exist

Publishing runs in CI via npm **Trusted Publishing (OIDC)** with provenance —
there is no long-lived npm token to steal, and packages publish in dependency
order with a preflight that aborts cleanly rather than half-shipping a release.
The full release process is documented in
[`agent-docs/RELEASING.md`](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/RELEASING.md)
and runs from [`publish.yml`](https://github.com/atomic-testing/atomic-testing/blob/main/.github/workflows/publish.yml).

### End-of-life is a written process that has already run once

Support isn't pulled by silence. MUI 5 / MUI-X 5 reached end of support on
2026-06-27 through a documented decision
([ADR-005](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/adr/005-drop-mui-5-support.md)),
with the packages frozen at `0.81.0`, deprecated in place (a `package.json`
marker and README banner — deliberately not an npm-registry deprecation, per
ADR-005), and later extracted with full history to a
[separate repo](https://github.com/atomic-testing/component-driver-mui-v5)
([ADR-014](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/adr/014-extract-mui-5-to-separate-repo.md)).
The support table lives in the
[README's stability & version support policy](https://github.com/atomic-testing/atomic-testing/blob/main/README.md#stability--version-support-policy);
the precedent is walked through on [API stability](./api-stability.mdx).

## Decisions are written down

Architecture decisions live in a public ADR log:
[`agent-docs/adr/`](https://github.com/atomic-testing/atomic-testing/tree/main/agent-docs/adr).
This is the paper trail an evaluator can audit — each record states the context,
the decision, and the alternatives that were rejected. Three load-bearing ones:

- [ADR-006](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/adr/006-1.0-api-freeze-and-evolution.md)
  — the 1.0 API freeze: what the stable surface is, how it's machine-checked,
  and how deprecations work.
- [ADR-007](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/adr/007-interactor-evolution-and-composition.md)
  — how the `Interactor` contract is allowed to grow after 1.0 without breaking
  third-party implementers.
- [ADR-015](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/adr/015-component-driver-protected-primitives.md)
  — narrowing `ComponentDriver`'s low-level primitives before the freeze,
  chosen explicitly because widening later is non-breaking and narrowing later
  is not.

### Mistakes are documented, not buried

The repo's own engineering guide contains a postmortem of a real cross-cutting
failure: a change to a shared interactor primitive (`pressKey`), made to fix one
driver, silently broke Angular Material's `MatSelect` and `MatAutocomplete` —
and slipped past a name-filtered local run — the break sat in the same package
whose `Select` suite was green — while per-major driver variants run as
separate CI jobs, so one green variant proved nothing about the rest.
The lesson (grep every call site, gate new behavior to the case that needs it,
run the full suites of affected consumers) is written into the
**"Shared-primitive blast radius"** section of the root
[CLAUDE.md](https://github.com/atomic-testing/atomic-testing/blob/main/CLAUDE.md),
where every future contributor — human or agent — reads it before touching a
shared primitive. A project that documents its failure modes in its own
onboarding file is telling you how it will behave when something breaks on your
watch.

The [ROADMAP.md](https://github.com/atomic-testing/atomic-testing/blob/main/ROADMAP.md)
is a manually synced summary of active work — it names the
[issue tracker](https://github.com/atomic-testing/atomic-testing/issues) as the
source of truth and admits it may lag, which is the honest framing for a
single-maintainer roadmap.

## Exit cost: what leaving looks like

The worst case for any single-maintainer dependency is abandonment. Here the
exit is cheap by construction, and this is worth stating once for the record:
every driver bottoms out in the same Testing Library and Playwright primitives
you would otherwise call by hand. Atomic Testing owns no renderer, no assertion
library, and no test runner. If the project stopped tomorrow, your tests keep
running against a frozen version, and migrating away means replacing driver
calls with the underlying Testing Library/Playwright calls — component by
component, at your own pace — not rewriting your component tree or your test
architecture. The argument is laid out in
[Why Atomic Testing](../why-atomic-testing.mdx).

## How to engage

- **Bugs and feature requests** — the
  [issue tracker](https://github.com/atomic-testing/atomic-testing/issues),
  using the provided templates.
- **Design-level proposals** — the
  [RFC template](https://github.com/atomic-testing/atomic-testing/blob/main/.github/ISSUE_TEMPLATE/rfc.md)
  for anything too large for a single issue: a new driver family, a breaking
  API change, a cross-cutting architecture change.
- **Contributions** — [CONTRIBUTING.md](https://github.com/atomic-testing/atomic-testing/blob/main/CONTRIBUTING.md)
  covers setup, required checks, the commit convention, and when a change needs
  an ADR.
- **Security vulnerabilities** — do **not** open a public issue; use GitHub's
  private vulnerability reporting from the repository's
  [Security tab](https://github.com/atomic-testing/atomic-testing/security), per
  [SECURITY.md](https://github.com/atomic-testing/atomic-testing/blob/main/SECURITY.md).

If you're weighing this page against a rollout decision, pair it with
[Adoption & rollout](./adoption-rollout.mdx) — the risk profile above is the
context; that page is the plan.
