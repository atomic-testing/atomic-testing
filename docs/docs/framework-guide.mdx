# Package Guide

:::tip Let the scaffolder pick for you
The fastest way to get exactly the right packages is [`create atomic-testing`](./quick-start.mdx) — it detects your framework, runner, and design system and installs the matching packages for you. This page is for hand-picking packages yourself or understanding what the scaffolder would choose; the [Framework and Runner Support](./support-matrix.mdx) matrix lists which combinations are verified vs. experimental.
:::

Every setup needs three things: a **framework/runner package** (the `createTestEngine` for your environment), the **core package**, and at least one **component-driver package** (the vocabulary for the components you interact with). Pick one from each group below.

## Choose your packages

| You have…                    | Install this framework package                                           |
| ---------------------------- | ------------------------------------------------------------------------ |
| React 19                     | `@atomic-testing/react-19`                                               |
| React 18                     | `@atomic-testing/react-18`                                               |
| React 17                     | `@atomic-testing/react-legacy`                                           |
| Vue 3                        | `@atomic-testing/vue-3`                                                  |
| Angular 20 / 21 / 22         | `@atomic-testing/angular-20`, `-21`, or `-22` (match your Angular major) |
| Any framework, in the browser | `@atomic-testing/playwright` (E2E) — pairs with any of the above         |

Then add the driver package for your UI library from [Component driver libraries](#component-driver-libraries) below. If you don't use a design system, `@atomic-testing/component-driver-html` covers standard HTML elements and works with any framework.

**DOM vs. E2E** is a runner choice, not an either/or: the framework packages run fast jsdom tests (Jest/Vitest), and `@atomic-testing/playwright` runs the same scene against a real browser. Most projects start with DOM tests and add Playwright for critical user journeys — the [tutorial](./getting-started-tutorial.mdx#8-same-scene-now-in-playwright-e2e) shows one scene reused across both.

For the exact `install` commands per framework (including peer dependencies), see [Manual Installation](./manual-installation.mdx); to have it done for you, run [`create atomic-testing`](./quick-start.mdx).

## Framework & runner notes

<Tabs>

<TabItem value="react" label="React">

`react-19`, `react-18`, and `react-legacy` (React 17) share an identical API — only the peer React version differs. On React 17, plan to upgrade: it's supported for gradual migration, not new work.

</TabItem>

<TabItem value="vue-3" label="Vue 3">

`VueInteractor` calls Vue's own `nextTick()` after each interaction, so reactive state settles before your next assertion — no `act()`-style flag to configure. Works with Composition-API and SFC-like components.

</TabItem>

<TabItem value="angular" label="Angular">

`@atomic-testing/angular-20`/`-21`/`-22` bootstrap a standalone component and await `ApplicationRef.whenStable()` after each interaction. They work under both zone.js and zoneless change detection (`zone.js` is an optional peer). Angular runs under **Vitest browser mode** — see the [support matrix](./support-matrix.mdx).

</TabItem>

<TabItem value="playwright" label="Playwright">

`@atomic-testing/playwright` drives a real browser (Chromium, Firefox, WebKit), so it covers cross-browser behavior, geometry, and anything jsdom can't model. It needs a page your app actually serves — point it at your dev server with `webServer`/`baseURL` in `playwright.config.ts`.

</TabItem>
</Tabs>

## Component driver libraries

Each driver package covers one UI library. Where a library ships several supported majors — Material UI (`component-driver-mui-v6`, `-v7`, `-v9`, plus the MUI-X equivalents) and Angular Material (`component-driver-angular-material-v20`, `-v21`, `-v22`) — **each major gets its own package**, because each major's DOM differs enough that a driver written against one won't reliably match another. The driver set itself (names, methods) is consistent across those majors. In practice: install and read the section matching the major you have, and skip the rest.

Each entry notes its framework binding and [support tier](./support-matrix.mdx) — **verified** (a green fixture proves it in this repo) or **experimental** (composed best-effort, no proving fixture yet). See the [generated API reference](/docs/api/component-drivers) for the full class and method list of any package.

### Material UI

_React · verified._ Match the package major to the MUI Core major you have installed.

<Tabs>
<TabItem value="mui-v9" label="MUI v9 (Latest)">

```bash

```

The full v7 driver set plus RadioGroup, Table, and Tooltip drivers. (MUI Core has no v8; v9 succeeds v7.)

</TabItem>

<TabItem value="mui-v7" label="MUI v7">

```bash
pnpm add @atomic-testing/component-driver-mui-v7
```

Core MUI v7 components with stable APIs.

</TabItem>

<TabItem value="mui-v6" label="MUI v6">

```bash
pnpm add @atomic-testing/component-driver-mui-v6
```

Core MUI v6 components with stable APIs.

</TabItem>
</Tabs>

**Support history:** MUI v5 / MUI-X v5 driver support ended (EOL) on 2026-06-27 under the published [versioning & EOL policy](./evaluate/api-stability.mdx). The `@atomic-testing/component-driver-mui-v5` and `@atomic-testing/component-driver-mui-x-v5` packages remain installable at their frozen final version, `0.81.0`, but receive no fixes, new drivers, or CI coverage. Rationale: [ADR-005](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/adr/005-drop-mui-5-support.md); the packages were later extracted with full history to a separate repo ([ADR-014](https://github.com/atomic-testing/atomic-testing/blob/main/agent-docs/adr/014-extract-mui-5-to-separate-repo.md)).

### HTML

_React & Vue 3 · verified; Angular · experimental._

```bash
pnpm add @atomic-testing/component-driver-html
```

Standard HTML elements (button, input, select, and so on). The right choice for custom components, HTML-based UI libraries, or when you need basic element interaction — and the safe default when no driver package exists for your design system.

## Tips

- **Start with HTML drivers and DOM tests** — they're faster to run and debug, and work with any components. Add a design-system driver package when you need library-specific behavior.
- **Mix driver packages freely** — HTML and MUI drivers can coexist in one scene.
- **Add Playwright gradually** — reuse the same scenes you already wrote for DOM tests against critical user journeys.
- **Upgrade drivers incrementally** — moving from `-v6` to `-v7` drivers is one import change, not a test rewrite.
