# Which Packages Do I Need?

:::tip Let the scaffolder pick for you
The fastest way to get exactly the right packages is [`create atomic-testing`](./quick-start.mdx) — it detects your framework, runner, and design system and installs the matching packages for you. This page is for hand-picking packages yourself or understanding what the scaffolder would choose; the [Framework and Runner Support](./support-matrix.mdx) matrix lists which combinations are verified vs. experimental.
:::

## 🤔 Choose Your Testing Strategy

### **DOM Testing (Fast, Unit-focused)**

Perfect for component testing, business logic, and rapid development cycles.

- **React Apps**: `@atomic-testing/react-18` or `@atomic-testing/react-19`
- **Vue Apps**: `@atomic-testing/vue-3`
- **Angular Apps**: `@atomic-testing/angular-20`, `-21` or `-22` (match your Angular major)
- **Vanilla JS**: `@atomic-testing/dom-core`

### **E2E Testing (Slow, Integration-focused)**

Great for user flows, cross-browser testing, and production-like scenarios.

- **Any Framework**: `@atomic-testing/playwright`

```bash
# Vue 3 with Material-UI (if using Vuetify/similar)
```

**Perfect for**: Vue 3 applications with custom components or HTML-based UI libraries

</TabItem>

<TabItem value="angular-html" label="Angular + HTML">

```bash

pnpm add @atomic-testing/angular-20 @atomic-testing/component-driver-html
```

**Perfect for**: Standalone-component Angular apps — works with both zone.js and

<TabItem value="e2e-any" label="E2E Testing (Any Framework)">

```bash
# E2E testing with Material-UI components
pnpm add @atomic-testing/playwright @atomic-testing/component-driver-mui-v9

pnpm add @atomic-testing/playwright @atomic-testing/component-driver-html

pnpm add @atomic-testing/playwright \
  @atomic-testing/component-driver-mui-v9 \
  @atomic-testing/component-driver-html
```

**Perfect for**: Cross-browser testing, user journey validation, production testing

</TabItem>

<TabItem value="fullstack" label="Full-Stack Testing">

```bash
# Cover all bases: DOM + E2E
pnpm add @atomic-testing/react-18 \
  @atomic-testing/playwright \
  @atomic-testing/component-driver-mui-v9 \
  @atomic-testing/component-driver-html
```

**Perfect for**: Comprehensive testing strategy with fast DOM tests + thorough E2E coverage

</TabItem>

## 🧩 Component Driver Libraries

Each driver-package section in the [generated API reference](/docs/api/component-drivers) covers one
UI library. Where a library ships several supported majors — Material UI
(`component-driver-mui-v6`, `-v7`, `-v9`, plus the MUI-X equivalents) and
Angular Material (`component-driver-angular-material-v20`, `-v21`, `-v22`) —
each major gets its **own package and its own generated section**, because
each major's DOM differs enough that a driver written against one won't
reliably match another.

The **driver set itself is consistent across those sections** — same driver
names, same methods — since they all document the same component-driver API
surface. In practice: install and read the section matching the major version
you have installed, and skip the rest.

Each entry below notes its framework binding and its [support tier](./support-matrix.mdx) —
**verified** (a green fixture proves it in this repo) or **experimental** (composed
best-effort, no proving fixture yet).

## 🤷 Not Sure Which to Choose?

### **Start Simple**

```bash
# If you're just getting started — react-19 is the default; on React 18, swap in @atomic-testing/react-18
pnpm add @atomic-testing/react-19 @atomic-testing/component-driver-html
```

You can always add more packages later. HTML drivers work with any components.

### **Decision Tree**

1. **What's your primary framework?**
   - React → `@atomic-testing/react-19` (on React 18, use `@atomic-testing/react-18` instead)
   - Vue → `@atomic-testing/vue-3`
   - Multiple/E2E → `@atomic-testing/playwright`

2. **What UI library do you use?**
   - Material-UI → `@atomic-testing/component-driver-mui-v9`
   - Custom/HTML → `@atomic-testing/component-driver-html`
   - Bootstrap/Other → Start with HTML drivers, build custom as needed

3. **What's your testing strategy?**
   - Fast unit tests → DOM testing packages
   - Integration tests → Add Playwright
   - Both → Install both!

## 💡 Pro Tips

- **Start with DOM testing** - It's faster and easier to debug
- **Add E2E tests gradually** - Focus on critical user journeys
- **Mix component drivers** - Use HTML + MUI drivers in the same project
- **Upgrade incrementally** - You can migrate from v6 to v7 drivers over time
