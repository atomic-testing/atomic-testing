# TypeScript 7 native (tsgo) LSP for Claude Code

Gives Claude Code real code intelligence on this monorepo ÔÇö go-to-definition,
find-references, find-implementations, and live diagnostics ÔÇö backed by the **same
engine the repo's `check:type` gate already uses** (`tsgo`, the Go-native TypeScript 7
compiler shipped as the `@typescript/native-preview` devDependency).

This is a self-contained, repo-local Claude Code plugin plus the scripts to enable it
in a Claude Code Cloud VM session.

## Why tsgo, and not `typescript-language-server`

The official `typescript-lsp` plugin drives `typescript-language-server`, which sits on
TypeScript's classic JS "Strada" language-service API. **TypeScript 7 / tsgo does not
expose that API** ÔÇö it speaks LSP natively through its own code path. So the stock
plugin can only give you navigation on _classic_ TypeScript, a different engine than the
`tsgo --noEmit` typecheck that actually gates CI here. This plugin points Claude Code
straight at `tsgo --lsp` instead, so navigation and diagnostics match CI.

> The invocation is `tsgo --lsp --stdio`. Note the binary is **`tsgo`**, not `tsc` ÔÇö even
> though `tsgo --help` self-identifies as _"tsc: The TypeScript Compiler."_ In this repo
> `pnpm exec tsc` resolves to classic TypeScript 6, which has no `--lsp` mode.

## Enabling it in a Cloud VM

**Preferred ÔÇö environment setup script** (Claude Code web UI Ôćĺ Environment settings Ôćĺ
Setup script). It runs pre-launch and is cached into the environment image, so the LSP
is live from session 1 with no plugin-load race. Paste the body of
[`setup-script.example.sh`](./setup-script.example.sh).

**Fallback ÔÇö SessionStart hook** (already wired in `.claude/settings.json`). Runs
`enable.sh` on every session start, but **only in the remote environment**
(`CLAUDE_CODE_REMOTE=true`) so it never mutates a local developer's global config. On the
_first_ session after adding it, the plugin installs but the LSP tool may not attach
until the next session (or `/reload-plugins`); it persists thereafter for the
environment's life.

**Manual** ÔÇö `bash tools/ts7-lsp/enable.sh` from anywhere in the repo.

### Preconditions

- **Claude Code Ôëą 2.1.205.** Below this, the `restartOnCrash` / `shutdownTimeout` fields
  in `.lsp.json` cause Claude Code to _silently skip_ the server. `enable.sh` warns if it
  detects an older build. (Verified working on the native binary at **2.1.211**; the
  "native binary can't do LSP" reports were against ÔëĄ2.1.15.)
- **Workspace trust.** Claude Code starts LSP servers only after the workspace is trusted
  ÔÇö accept the trust prompt in the interactive session.
- **`tsgo` present.** It ships via `pnpm install` (`@typescript/native-preview`).
  `enable.sh` warns if `pnpm exec tsgo` isn't resolvable.
- **Built `dist` for cross-package navigation** ÔÇö see below.

### Acceptance test

```bash
pnpm --filter @atomic-testing/core build      # cross-package targets live in dist
python3 tools/ts7-lsp/verify-navigation.py    # asserts definitions resolve
```

Then in an interactive session: confirm the LSP tool is listed, and ask Claude for the
definition of an imported `@atomic-testing/*` symbol.

## Cross-package navigation (a real limitation, and the upgrade path)

Empirically (via `verify-navigation.py`):

| Navigation | Resolves to |
| --- | --- |
| **Within a package** (relative import) | the real source in `src/` Ôťô |
| **Across packages** (`@atomic-testing/*`) | the bundled `dist/*.d.mts` declaration file |

Cross-package jumps land in the built declaration file, not source, because each
package's `exports` map points at `dist/`. Two consequences:

- **`dist` must be built** or cross-package imports have no jump target at all (the same
  stale-`dist` trap documented in the root `CLAUDE.md`).
- **`declarationMap` will _not_ fix this.** The `.d.ts` is generated by `tsdown`'s bundler
  (rolldown), which emits JS sourcemaps but no declaration maps ÔÇö so `declarationMap: true`
  in `tsconfig` is a no-op here.

**Upgrade path (deliberately a separate change):** a `tsconfig` `paths` mapping
(`@atomic-testing/*` Ôćĺ `packages/*/src/index.ts`) makes cross-package go-to-definition
land in real source, verified via `verify-navigation.py`. It is **not** bundled into this
enablement because it also feeds `tsgo --noEmit`, changing `check:type` resolution across
every package and per-major variant ÔÇö that needs its own full typecheck validation. Note
for whoever picks it up: TypeScript 7 **removed `baseUrl`** and requires relative path
targets (`./packages/*/src/index.ts`), so the mapping must be written in the TS7-compliant
form.

## Troubleshooting

| Symptom | Cause / fix |
| --- | --- |
| No LSP tool in the session | Workspace not trusted, or plugin installed this session (run `/reload-plugins` or restart). |
| Server silently absent | Claude Code < 2.1.205 skipping the restart fields ÔÇö upgrade, or drop those two fields. |
| `Executable not found` | `pnpm install` hasn't run; `tsgo` isn't resolvable. |
| Cross-package jumps go nowhere | `dist` not built ÔÇö `pnpm --filter @atomic-testing/core build`. |
| LSP works on native binary? | Yes, verified at 2.1.211. Older (ÔëĄ2.1.15) native builds had a gap. |
